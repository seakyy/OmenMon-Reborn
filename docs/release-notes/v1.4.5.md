# v1.4.5 — EC read-path hardening + field-report sweep

## OmenMon-Reborn v1.4.5

**EC read-path hardening + field-report sweep.** This release attacks the root
cause of the recurring "RPM = negative / millions" and bad-temperature reports
(#86) at the Embedded Controller read path itself, rather than patching board by
board — plus a threading/safety sweep, a more robust auto-detector, live fan
telemetry in `-Diag`, and a batch of model-database updates.

### 🩹 Fixed

- **Garbage bytes after the EC wait fail-limit (#86).** A sustained EC wait
  failure no longer returns a stale/garbage Data byte — it fails cleanly so the
  retry wrapper keeps the last good value. This is the most likely common root
  cause of the negative/huge-RPM and bad-temperature reports.
- **Torn 16-bit reads (#86).** `ReadWord` now requires two consecutive identical
  reads before trusting a value, eliminating self-inconsistent words from an EC
  state change between the low/high byte transactions.
- **EC lock churn on the monitoring path (#86).** Each monitor tick now reads the
  whole sensor array under a single mutex hold (`Hw.EcExecBatch()`) instead of
  taking/releasing the cross-process lock once per sensor.
- **EC wait backoff could exceed the mutex timeout (#88).** `Wait()` now yields
  the timeslice before escalating to a 1 ms sleep, cutting the worst-case
  per-`Wait` hold from 25 ms to 15 ms so a Word read can no longer pin the EC
  mutex past its timeout.
- **EC lifecycle race + `gpuTempObserved` data race.** Init/Close now run under a
  per-singleton lifecycle lock; the observed-GPU-temp flag is `volatile`.
- **`NullReferenceException` in `NvMuxGetState()` on non-Optimus systems.**
- **Thermal-panic protection no longer tied to the dynamic tray icon** — over-temp
  protection now runs on every icon tick when enabled, regardless of icon mode.
- **8A18 (OMEN 17 ck1000nw) post-calibration fan lock (#84).**
- **GPU fan row showed garbage on single-fan boards (#81, 8BB3).**

### ✨ Added

- **More robust EC layout auto-detection (#37)** — read-only and conservative; no
  longer misfires on idle fans / off-`0xB0` tachometers.
- **Live Fan Telemetry section in `-Diag` (#49)** — per-fan level, duty-cycle,
  resolved speed, and the exact register/mode/multiplier used, so a wrong RPM is
  immediately traceable.
- **Native HP Omen Max 16 (8D41, 2025) entry (#87, #90)** — full `<Model>` entry
  with confirmed 16-bit LE tachometers.
- **HP OMEN 16 wd0xxx (8BA9, 2024) read-only RPM mapping (#92)** — single LE16
  tachometer at `0xF1`, flagged `SingleFan`; RPM displays correctly out of the box.
- **Omen 16-am1001nw coverage (#31)** — covered by the existing `8E71` entry.
- **`EcWaitYieldCount` config knob** in `OmenMon.xml`, plus EC-timing docs and an
  RPM/temperature troubleshooting section (#14).

### ⏳ Deferred (need confirmed data before a safe code change)

- HP Omen 16-wd0004nw (#51/#75), xd0020ax (#77), and the Ryzen 9 7945HS SKU (#85)
  remain on the read-only auto-detector + Auto-Calibration wizard. Promoting them
  needs a confirmed Product ID + register dump — see "Pending field reports" in
  `wiki/Model-Database.md`.
