# v1.4.6 — UI responsiveness + CLI restoration + field-report sweep

## OmenMon-Reborn v1.4.6

**UI responsiveness + CLI restoration + temperature-policy fix.** This release
finishes what the v1.4.5 threading sweep started: all periodic hardware access
now runs on a dedicated background monitor thread, so the tray menu and the
main window can never again stall behind a contended EC mutex (#98). It also
restores the entire command-line interface (#101), fixes fan curves being
pinned by a stuck EC temperature byte (#97), silences the recoverable EC-lock
error boxes (#94), and adds two boards to the model database.

### 🩹 Fixed

- **CLI produced no output for any argument (#101).** A v1.4.2 redirection
  guard misclassified "no console handles" (the normal state of a GUI app
  started from cmd/PowerShell) as "output redirected" and never attached the
  console. `-Bios`, `-Ec`, usage help — everything prints again; redirecting
  output to a file (`-Diag > report.md`) still works.
- **Sluggish tray menu / "teleporting" window drag since v1.4.5 (#98).** New
  `GuiMonitor` background thread owns all periodic sampling and control
  (sensors, fan-program tick, thermal panic, countdown maintenance, BIOS
  heartbeat) and publishes immutable snapshots the UI renders lock-free. The
  UI thread no longer performs any EC/BIOS I/O during passive operation —
  right-clicking the tray icon requests a fresh sample off-thread and renders
  from the snapshot.
- **CPU temperature stuck at a constant value pinned the fan curve (#97,
  8D87).** CPU temperature is now the *higher* of the EC CPUT register and the
  WMI BIOS sensor, so a CPUT byte stuck on firmware string data (8D87 reads a
  permanent 52) can no longer hold fan programs at one curve row while the die
  overheats. The policy only ever errs towards more cooling.
- **"Failed to acquire embedded controller exclusive lock" boxes (#94).**
  EC-mutex timeouts on periodic/recoverable paths (monitor, startup
  AutoConfig, calibration sweep) now skip-and-retry silently instead of
  raising a modal box; user-initiated actions stay loud. A new "EC lock
  timeouts this session" row in `-Diag` keeps the contention visible in field
  reports.
- **Calibration pause race** — after the wizard pauses the monitor, no
  trailing monitor pass can slip in anymore.

### ✨ Added

- **HP Victus 16-e0xxx (88ED, 2022) native model entry (#99)** — classic 2022
  layout with LE16 tachometers at `0xB0`/`0xB2`, field-tested by the reporter.
- **HP Victus 15-fb0102la (8A3E, 2023) read-only RPM mapping (#96)** — LE16
  tachometers at `0xB0`/`0xB2` pinned in `AutoCal.KnownBoards`; control stays
  on the proven default layout.
- **`-Diag`: EC lock timeout counter** in the Kernel Driver section.
- **Model coverage notes** — 8787 / Omen 15 2020 (#95) confirmed on the
  default layout (`GetFanLevel` BIOS error on that generation is expected and
  harmless); Victus 16-s1084AX (#100) documented as a pending second hardware
  variant behind `8C9C` — see `wiki/Model-Database.md`.

### ⏳ Deferred (need confirmed data before a safe code change)

- **8D87 `TempCpuReg` override (#97)** — the max(CPUT, BIOS) policy unblocks
  fan curves now; mapping the board's real EC temperature register needs a
  `-Probe` dump under load from the reporter.
- **#100 (s1084AX) `0xD6`/`0xD8` tachometers** — needs a `-Diag` Product ID
  confirmation before touching the shipped `8C9C` mapping (HP recycles product
  IDs across regional variants); the reporter's wizard sidecar already covers
  their machine locally.
